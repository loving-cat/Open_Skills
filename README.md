# 专家团模式 (Expert Team Mode)

> Claude Code 多Agent协作框架 - 并行执行、记忆持久化、跨会话恢复

## 功能特性

| 特性         | 说明                                   |
| ------------ | -------------------------------------- |
| 多Agent并行  | 前端/后端/数据库/DevOps 专家同时工作   |
| 任务队列驱动 | 自动拆解任务，智能分配专家             |
| 记忆持久化   | 工作状态自动保存，支持跨会话恢复       |
| 持续协作     | 完成后自动询问下一步，不会主动结束对话 |
| 自动确认     | 日常操作自动执行，仅重大节点需确认     |

## 安装配置

### 1. 复制文件

将解压后的文件复制到 Claude Code 配置目录：

```
~/.claude/
├── CLAUDE.md           ← 复制到此处（覆盖或新建）
└── skills/
    └── Think/          ← 复制整个目录到此处
        ├── skill.md
        ├── config/
        ├── templates/
        └── utils/
```

### 2. 目录结构

```
~/.claude/
├── CLAUDE.md                    # 全局配置（专家团模式）
├── skills/
│   └── Think/                   # 专家团技能
│       ├── skill.md             # 主入口
│       ├── config/
│       │   ├── experts.yaml     # 专家角色配置
│       │   ├── tools.yaml       # 工具规范配置
│       │   └── workflow.yaml    # 协作流程配置
│       ├── templates/           # 专家 Prompt 模板
│       │   ├── frontend.md
│       │   ├── backend.md
│       │   ├── database.md
│       │   ├── devops.md
│       │   ├── security.md
│       │   ├── test.md
│       │   └── common.md
│       └── utils/
│           └── prompt-builder.md
└── projects/
    └── {project}/
        └── memory/
            ├── MEMORY.md        # 记忆索引
            └── team-state.md    # 工作状态
```

### 3. Windows 路径

```
C:\Users\{你的用户名}\.claude\
├── CLAUDE.md
└── skills\Think\
```

## 使用方法

### 启动专家团

**方式一：使用命令**

```
/Think
```

**方式二：直接描述需求**

```
帮我创建一个登录系统
```

专家团会自动启动。

### 恢复上次工作

```
/Think 继续上次的项目
```

## 专家团队

| 专家       | 人数 | 职责                         | 触发任务              |
| ---------- | ---- | ---------------------------- | --------------------- |
| 前端团队   | 3人  | UI组件、页面开发、样式实现   | 页面、组件、样式      |
| 后端团队   | 3人  | API开发、业务逻辑、服务实现  | API、接口、业务逻辑   |
| 数据库团队 | 2人  | 数据模型、迁移脚本、查询优化 | 表设计、SQL优化、迁移 |
| DevOps团队 | 2人  | Docker配置、CI/CD、部署脚本  | Docker、部署、CI/CD   |
| 安全专家   | 1人  | 安全审计、漏洞修复           | 安全、漏洞、权限      |
| 测试专家   | 1人  | 测试验证、质量检查           | 测试、验证            |

## 协作流程

```
Phase 1: 规划       → 架构师输出 Plan + 任务分解
Phase 2: 分配       → 根据任务类型分配专家
Phase 3: 执行       → 波次并行调度（无依赖同时启动）
Phase 4: 增援       → 疑难问题自动调度专家
Phase 5: 验收       → 测试专家功能验证
```

### 并行执行示例

```
用户需求: 创建一个登录系统

Plan 输出:
┌─────────────────────────────────────────────┐
│ Task1: 用户表设计     → 数据库专家          │
│ Task2: 认证API开发    → 后端专家            │
│ Task3: 登录页面组件   → 前端专家            │
│ Task4: 集成测试       → 测试专家            │
└─────────────────────────────────────────────┘

执行顺序:
波次1: Task1, Task2, Task3 同时启动 (并行)
波次2: Task4 等待 Task1-3 完成后启动
```

## 自动确认机制

| 操作类型                         | 行为                   |
| -------------------------------- | ---------------------- |
| 写代码、改文件、跑测试           | ✅ 自动执行             |
| 小任务（改函数、修bug）          | ✅ 自动执行             |
| 中等任务（新功能模块）           | ✅ 自动执行（简要说明） |
| 大任务（新系统）Plan完成         | ⏸️ 需确认               |
| 关键决策（技术选型、架构变更）   | ⏸️ 需确认               |
| 风险操作（删除、部署、强制推送） | ⏸️ 需确认               |

## 配置说明

### 专家配置 (config/experts.yaml)

```yaml
experts:
  frontend:
    name: 前端专家
    count: 3              # 可调度人数
    skills:
      - React/Vue/Angular 开发
      - TypeScript 类型系统
    triggers:
      - UI组件开发
      - 页面开发
    template: templates/frontend.md
```

### 流程配置 (config/workflow.yaml)

```yaml
workflow:
  phases:
    - id: plan
      name: 规划
      role: 架构师
    - id: execute
      name: 专家执行
      parallel_rule: 无依赖任务同时启动 Agent
```

### 工具配置 (config/tools.yaml)

定义可用的代码编辑工具和搜索工具。

## 记忆持久化

### 自动保存时机

| 事件      | 更新内容                         |
| --------- | -------------------------------- |
| Plan 完成 | 写入技术选型、架构摘要、任务分解 |
| 任务开始  | 更新任务状态为 `in_progress`     |
| 任务完成  | 更新任务状态为 `completed`       |
| 项目完成  | 更新 `Plan 状态: completed`      |

### 记忆文件位置

```
~/.claude/projects/{project}/memory/team-state.md
```

## 示例

### 示例1：创建新系统

```
用户: 创建一个用户管理系统

专家团:
1. 输出 Plan + 技术选型
2. [确认]
3. 并行执行:
   - 数据库专家: 设计用户表
   - 后端专家: 实现用户 CRUD API
   - 前端专家: 开发用户管理页面
4. 测试专家: 验收测试
5. "还有其他任务吗？"
```

### 示例2：代码审查

```
用户: 检查 src/ 目录的代码质量问题

专家团:
1. 调度代码审查专家
2. 输出问题报告
3. 询问是否修复
4. "还有其他任务吗？"
```

### 示例3：小任务

```
用户: 帮我改一下 main.cpp 的变量名

专家团:
1. 直接修改
2. "还有其他要改的吗？"
```
